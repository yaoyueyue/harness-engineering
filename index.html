Harness Engineering

Agentic engineering, not prompt engineering — a reusable scaffold that automates engineering work end-to-end, so the system (not the prompt) owns the loop.

What it is Lifecycle Components Human checkpoints Design principles

What this is

The shift is from prompt engineering to agentic engineering — building the scaffold the agent runs inside (specs, templates, scripts, state files, human checkpoints).

With the scaffold, automation becomes repeatable, parallel, and auditable. The agent isn't asked to remember conventions, decide where to write files, or invent a workflow each session — the harness provides all of that. The agent's job is the part agents are actually good at: executing one well-defined step at a time. Originally built to automate a multi-feature CLI migration; the scaffold is the reusable part.

The lifecycle — six states, three agents

Each feature moves through six states. Every transition is owned by exactly one agent, and the work inside each state is itself a sub-loop that can be invoked independently — you can run just the spec writer or just the open-PR sweep without going through the whole lifecycle.

Components — the stack, top to bottom

Seven layers. Higher layers dispatch to lower ones; lower layers feed iteration back up. Every name below is a real file in the repo.

  1. CLAUDE.md — entry point. Dispatches the agent to the right docs by situation (implementing, validating, PR setup, etc.).
  2. Agent files — .claude/agents/{orchestrator, porter, validator}.md. Role, tools, never-dos. One file per agent.
  3. Skills — skills/{lifecycle, porter_phase1/2/3, pr_check, pr_setup, validator/runbook}.md. Step-by-step playbooks for each phase. The agent reads only the skill it's currently in.
  4. Templates — templates/{spec, test, plan, diff_report, pr_standard, lifecycle}.md. Frozen output formats — every artifact looks the same so the next agent can parse it without thinking.
  5. Specs — specs/{id}/{spec, test, diff_report}.md + code_hash + logs/. Per-feature plan and evidence. The unit of work.
  6. Scripts — scripts/{render_dashboard.py, validator_lib.sh, setup_validator.sh, resolve_bot_comments.py, update_feature.py, ...}. Deterministic background jobs. If you can write it as code, it shouldn't be an agent.
  7. Iteration files — human.md (human → Orchestrator inbox) · needs_human_review.md (agent → human escalation) · harness.md (open infra issues, H-IDs) · todo.md · tomorrow.md · featurelist.json (state-machine truth). Where the system evolves between sessions.
How a new agent finds its way

Always starts at layer 1 (CLAUDE.md) and walks down only as far as it needs. Porter writing a spec? CLAUDE.md → porter.md → porter_phase1.md → templates/spec.md → writes to specs/{id}/spec.md. No layer is loaded unless needed.

Human checkpoints

Agents execute. Humans judge — and judgment can't be replaced. Execution can be codified, scripted, automated. Judgment can't. The three checkpoints below are exactly the calls where good judgment matters more than fast execution, and they stay with humans by design.

Frame

Humans never have to ask "what is the agent doing right now?" — that's what state files and structured logs are for. The harness tells them exactly which decision is needed and where the evidence lives (which specs/{id}/test.md, which PR comment, which landing-order question).

Design principles

Four principles encoded into the repo's structure that keep the harness from collapsing under its own weight:

Progressive disclosure

CLAUDE.md dispatches to a phase doc, which references a template, which produces a log. Each layer expands only when needed. Rules live once in RULES.md; other docs reference rule IDs (VAL-001, REV-001) instead of restating them.

Traceable logs & H-IDs

Every recurring failure pattern gets an H-id in harness.md (H041, H062, …). Validator's Pattern Recognition step explicitly checks: does this FAIL match an existing H-id? If so, reference it; if not, create one. The harness's mistakes don't repeat.

Code & harness stay separate

The harness lives in its own git repo on its own branch. Feature branches contain ONLY plugin code + tests + BUILD files. Reviewers see only the code change.

State in files, not in agent heads

Any agent resumes from cold by reading featurelist.json + spec.md + code_hash. Restarts cost nothing. The harness survived a full migration of its own location (in-tree → standalone worktree) without losing in-flight features.

Take-away

Harness engineering turns "agents that can sort of do stuff" into autonomous delivery you'd trust in production. The agents are the easy part — the scaffolding (specs, templates, scripts, state files, named human checkpoints, encoded H-IDs) is what makes the system reliable, debuggable, and reusable.

Borrow whichever pieces fit. The components are independent by design: take just the layered file hierarchy, just the Validator log convention, just the H-ID encoding, or just the human.md drain — each works on its own.
